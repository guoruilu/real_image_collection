# 技术路线详解

## 1. 总体思路
"下载干净的真实物体照片"这件事看起来简单，但有三个不平凡的子问题：

1. **召回**：怎么从互联网拿到足够多的候选图？（搜索引擎结果质量参差不齐，单一源召回率低）
2. **过滤**：怎么剔除卡通、表情包、3D 渲染、手办、绘画——这些都不是"真实物体照片"？
3. **裁剪**：怎么精确定位目标在图中的位置，把背景去掉？且目标类别可能不在常规检测数据集（COCO）的 80 类里。

我们用**三段式**对应解决，每段挑当前最合适的模型，整体兼顾零样本通用性、精度、显存预算（8 GB GPU）：

```
[多引擎多语种爬取]  →  [CLIP 双重过滤]  →  [YOLO-World 开放词表检测裁剪]
   召回                  类别正确 + 真实照片         精确定位 + 任意类别
```

## 2. 各组件角色与选型理由

### A. 爬取层：`icrawler` (Bing + Baidu)
- **角色**：宽召回，多语种、多平台覆盖，确保候选池足够大让后续过滤有余量。
- **为什么选 icrawler**：免 API key、支持多搜索引擎、可控线程数和最小尺寸；相比 `google-images-search` 等更稳定，不依赖代理。
- **为什么要 Bing + Baidu 并行**：经验上 Bing 对英文术语好（如 "Egyptian Mau"），Baidu 对中文术语召回率明显更高（"埃及猫" 上 Baidu 一关键词能拉 40 张，Bing 常只有 1~3 张）。两者互补。
- **为什么不用 Google**：icrawler 内置 Google 需要稳定海外网络，环境不一定具备；Bing+Baidu 已经够用。
- **为什么不用 LAION / 已有数据集**：本任务需要任意细类（如"埃及猫品种"），现成数据集多按粗类组织，找细类要重检索；爬取更直接，且包含更多近期照片。

### B. 真实性 + 细类过滤层：CLIP-ViT-L/14（OpenAI 权重）
- **角色**：
  1. **真实性二分类**：给每张图打分"是真实照片，还是卡通/插画/3D 渲染/手办/绘画/动漫……"
  2. **细类辨别（可选）**：给每张图打分"是目标子类（如埃及猫），还是常见混淆子类（波斯猫/布偶/暹罗……）"
- **为什么选 CLIP**：
  - **零样本**：换类别只改 prompt，不用重训分类器——这是项目通用化的关键。
  - **图文双塔**：可以一次性把"a photo of X" / "a cartoon of X" 这种细粒度语义对比做到位，传统纯视觉分类器做不到。
  - **OpenAI 权重 vs LAION**：OpenAI 权重在"照片 vs 非照片"这种风格判别上更稳，LAION 权重更偏概念匹配。
- **为什么选 ViT-L/14 而非 B/32**：
  - L/14 在风格/材质判断（绘画感、动漫线条、3D 高光）上明显更准
  - 显存 ~2.5 GB，4070 完全吃得下
  - B/32 速度快但负样本（卡通/手办）误判率明显更高，过滤之后还会残留噪声
- **为什么不用专门的 "AI/cartoon detector"**：那类模型通常只针对一种伪造（如 GAN/diffusion 生成图），无法覆盖"手办照片"这种真实物体但非目标的边缘情况。CLIP 的开放语义最适合定义"我想要什么、不想要什么"。
- **打分细节**：
  - 真实性：取多个正模板和多个负模板的均值相似度，再用温度 100 做 softmax，得到 `P(real)`
  - 细类：拼接正/负 prompt 一起 softmax，取正类概率质量之和

### C. 检测裁剪层：YOLO-World v2-s
- **角色**：在已过筛的候选图上，**用一段文字描述**框出目标位置，按框 + 5% 边距裁剪。
- **为什么需要开放词表检测**：
  - 传统 YOLO（COCO 80 类）覆盖不了"熊猫""特定乐器""特定鸟类"等很多日常类
  - 即使目标在 COCO 类里，开放词表也允许细化（如 "acoustic guitar" vs 仅 "guitar"）
- **为什么选 YOLO-World 而非 OWL-ViT / Grounding DINO**：

  | 模型 | 推理速度 | 精度 | 显存 | 备注 |
  |---|---|---|---|---|
  | YOLO-World-s | 快（~30 ms/张） | 中上 | <1 GB | 与 ultralytics 一体，接口简洁 |
  | OWL-ViT base | 中（~150 ms） | 中 | ~2 GB | HuggingFace 调用稍重 |
  | Grounding DINO | 慢（~400 ms） | 高 | ~3 GB | 装包麻烦，CUDA 算子有兼容性 |

  本任务是"批量过筛"，**速度优先**，YOLO-World 性价比最高。要更准可换 `yolov8l-worldv2.pt`（仍是 YOLO-World 系列）。
- **关键经验：检测词要用通用类，不要用细类**：
  - 试过 `--detect "egyptian mau cat"`：YOLO-World 召回从 49/50 掉到偶有失败
  - 改用 `--detect "cat"`：召回稳定 50/50
  - 原因：细类描述把开放词表分类器拉到稀有训练样本区；通用类（"cat"）落在主流分布上，定位最稳。**细类判断在 CLIP 那一步已经做了**，检测层只负责"定位"，不应再承担"分类"。

## 3. 为什么是这个组合，而不是别的
- **替代方案 1：纯 YOLO + 人工筛**——能完成定位但解决不了卡通/品种问题，且 COCO 类受限。
- **替代方案 2：CLIP 分类 + 显著性检测裁剪**——显著性检测对多物体、复杂背景不稳，裁出来的框边界不齐。
- **替代方案 3：Segment Anything 抠图**——SAM 是无监督分割，需要 prompt（点/框），还是要先用开放词表检测器给出框，等于多一步。
- **当前组合**：CLIP 把"风格/类别"问题用语义解决，YOLO-World 把"定位"问题用快速检测解决，各管各的，零样本通用。

## 4. 模型间显存调度
两个大模型不同时常驻，节省显存：
1. 加载 CLIP-L，跑完所有候选图的过滤，**`del model + torch.cuda.empty_cache()`**
2. 再加载 YOLO-World 做检测裁剪

这样峰值显存约 3 GB，余量足够跑其它进程。
